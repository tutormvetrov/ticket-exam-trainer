# Mac Build Smoke CI — Design

**Дата создания:** 2026-04-20
**Цель:** обеспечить надёжную проверку macOS-сборки до тега релиза, чтобы на `v*`-пушe не всплывали ошибки сборки впервые.

---

## 1. Motivation

Текущий `.github/workflows/release.yml` собирает `.app` на `macos-latest` только на пуш тега `v*`. До этого момента никакая Mac-специфичная проблема не ловится:

- `ci.yml` гоняет pytest на `ubuntu-latest` и `windows-latest` — macOS отсутствует;
- разработка ведётся на Windows, `build_mac_app.sh` запускается впервые только в release-workflow;
- если `flet pack`, `pip install` или проверка seed падает на Mac — узнаём об этом уже после пуша тега, с красным релизом в истории.

Нужен способ **заранее** (до тега) проверить, что Mac-сборка проходит. Без автотриггера на каждый push/PR (экономим Mac CI-минуты), но с чёткой точкой входа.

## 2. Decisions

| Параметр | Выбор | Отклонённые варианты и почему |
|---|---|---|
| Триггер | **`workflow_dispatch` only** (ручной запуск из Actions UI) | Авто на push в main — гонка на каждый коммит, расход Mac-минут; PR + push — ещё дороже, блокирует PR-мердж на 15 мин; cron — лишний шум |
| Runner | **`macos-latest`** | Фиксация на `macos-14` / `macos-15` — нет нужды, latest достаточно для smoke |
| Глубина проверки | **α — только сборка** (build + проверка наличия `.app`) | β (запуск бинаря на 15с) и γ (проверка app_data после запуска) — пользователь scope сузил до «чтобы собирался билд качественно», рантайм-поведение не в рамках этой задачи |
| Артефакты | **Не загружаем** | Загрузка `.app` — семантика релиза, не smoke. Если нужен тестовый бандл — делается отдельным `v*-rc*` тегом через существующий `release.yml` |
| pytest на macOS | **Не добавляем в этом изменении** | Пользователь явно выбрал вариант B (build smoke), не A (pytest matrix). YAGNI — расширение матрицы возможно отдельной задачей |
| Интеграция в workflow релиза | **Остаётся независимым workflow** | Инлайнинг в `release.yml` не даёт pre-tag-проверку — суть задачи именно в том, чтобы запускать ДО тега |

## 3. Architecture

Один новый файл: `.github/workflows/mac-build-smoke.yml`. Копирует структуру job'а `build-macos` из `release.yml`, но:

- триггер заменяется на `workflow_dispatch`;
- удаляются шаги `Package macOS zip` и `Upload macOS artifact`;
- добавляется явная пост-проверка `dist/Tezis.app` (на случай, если `build_mac_app.sh` в будущем перестанет сам валидировать).

### 3.1 Workflow файл (псевдокод шагов)

```yaml
name: Mac build smoke

on:
  workflow_dispatch:

permissions:
  contents: read

jobs:
  build:
    runs-on: macos-latest
    steps:
      - checkout@v6
      - setup-python@v6 (3.12, cache: pip)
      - pip install -r requirements.txt && pip install pyinstaller
      - verify data/state_exam_public_admin_demo.db exists
      - chmod +x scripts/build_mac_app.sh
      - ./scripts/build_mac_app.sh python3 data/state_exam_public_admin_demo.db
      - verify dist/Tezis.app directory exists
```

Никаких переменных окружения, secrets, cache'ей сверх того, что уже есть в `release.yml` для аналогичного job'а.

### 3.2 Что НЕ входит

- Автотриггер на push/PR — сознательно опущен (см. Decisions).
- Запуск собранного бинаря — опущен (α, не β).
- Publishing artifacts — опущен (не релиз).
- Notifications (Slack/email на падение) — опущены (пользователь смотрит в UI Actions сам перед релизом).
- Matrix по версиям macOS (`macos-13`, `macos-14`) — опущено, `macos-latest` достаточно для smoke.

## 4. Applied workflow (как этим пользуется пользователь)

Перед каждым пушем релизного тега `v*`:

1. Открыть GitHub → вкладка **Actions** → слева выбрать **Mac build smoke**.
2. Нажать **Run workflow** (справа сверху), выбрать ветку (обычно `main`), подтвердить.
3. Подождать ~10-15 минут.
4. Если зелёный ✓ — Mac-сборка работает, можно пушить тег.
5. Если красный ✗ — смотреть логи в том же job'е, чинить, повторять.

Правило закреплено в memory (`feedback_release_macos.md`) — будет всплывать в будущих сессиях при подготовке релиза.

## 5. Testing

**Ручное тестирование на стороне пользователя:**
1. Запустить workflow через UI на текущем main — убедиться что зелёный.
2. (Опционально) Временно сломать `build_mac_app.sh` (например, добавить `exit 1` в начало) — убедиться что workflow становится красным и причина видна в логах. Откатить.

**Автоматическая проверка:** не требуется — смоук сам по себе и есть проверка.

## 6. Risks & Mitigations

| Риск | Смягчение |
|---|---|
| `flet pack` ломается на конкретной версии `macos-latest` после апдейта раннера | Пользователь увидит это при запуске перед релизом и либо починит, либо временно зафиксирует runner на `macos-14` |
| Пользователь забыл запустить workflow перед тегом | Memory-запись `feedback_release_macos.md` напомнит в будущих сессиях; в README релизной процедуры можно добавить чек-лист |
| `data/state_exam_public_admin_demo.db` отсутствует в репо | Workflow падает на шаге verify и даёт понятную ошибку — поведение желаемое |
| Mac-минуты CI кончаются | При ручном триггере это невозможно — каждый запуск явный, ~15 мин за раз, в бесплатном лимите (2000 мин / 10x = 200 Mac-мин = ~13 запусков/мес) — запас кратный |

## 7. Follow-ups (не в этой задаче)

- Добавление `macos-latest` в pytest-матрицу `ci.yml` — если обнаружатся Mac-специфичные рантайм-баги вне сборки.
- Переход на рантайм-смоук (β): реальный запуск `Tezis.app/Contents/MacOS/Tezis` с таймаутом 15с — если пользователь столкнётся с проблемами, которые сборка не ловит.
- Чек-лист релиза в `docs/` или `README_classmates.md` — документированная процедура со ссылкой на workflow.
