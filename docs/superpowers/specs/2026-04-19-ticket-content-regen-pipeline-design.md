# Регенерация контента билетов через Claude API — design

## Контекст и проблема

В БД `exam_trainer.db` 208 билетов госэкзамена ГМУ. У каждого билета 6 блоков: INTRO / THEORY / PRACTICE / SKILLS / CONCLUSION / EXTRA. На этих блоках работают 5 режимов тренировки.

QA-сверка с исходным PDF (`МДЭ_ГА_2024_Кол_Конспект_ГМУ_ГАРФ_18_02_2025_в_2.pdf`, 168 страниц, 26 разделов × 8 билетов):

- **76 % билетов** имеют тело в PDF и в принципе восстановимы из исходника.
- **23 % билетов** в исходнике пустые — текст был дописан в другой Claude-сессии.
- Содержательного «мусора» при текущем построении нет.
- Но **типизация блоков не соблюдается**: текущий парсер `application/answer_block_builder.py` разрезает один параграф на 4–6 кусков по эвристике и подписывает их `**Пример N.**` / `**Фрагмент N.**` / `**Основное содержание.**`. Получается формально 6 блоков с одинаковым по типу контентом — не «теория, потом практика, потом навыки», а просто соседние куски одного абзаца.

Пример (tkt-002, «Выборы в федеральные органы…»):

```
INTRO    — описание трёхуровневой системы выборов (по сути это и есть введение)
THEORY   — "**Основное содержание.**" + дубликат INTRO
PRACTICE — "**Фрагмент 2.**" + продолжение того же текста
SKILLS   — "**Фрагмент 3.**" + следующий кусок
CONCLUSION — повтор последних абзацев INTRO без «Таким образом»
EXTRA    — "**Фрагмент 4.**" + одно оборванное предложение
```

Цель — каждый блок **по типу и объёму** соответствует структуре идеального ответа на отлично, описанной в методичке `Структура_ответа_МДЭ_на_отлично_ГМУ.pdf`.

## Эталонная структура блоков (по методичке МДЭ)

### INTRO — Введение (постановка проблемы)
- Краткое описание вопроса/проблемы билета.
- Указание на актуальность темы в контексте ГМУ.
- Формулировка цели ответа.
- **Объём:** 80–150 слов, 2–3 абзаца.

### THEORY — Теоретическая часть (знать)
- Основные концепции, подходы, теории.
- Нормативно-правовая база: законы, постановления, регламенты с номером и годом.
- Ключевые термины и определения.
- Ссылки на научные источники.
- **Объём:** 180–280 слов. Обязательно ≥1 нормативный акт, ≥2 термина с определением.

### PRACTICE — Практическая часть (уметь)
- Анализ ситуации/проблемы.
- Конкретные меры, методы, инструменты решения.
- Обоснование выбранных подходов.
- Примеры из практики ГМУ (российской и/или международной).
- **Объём:** 180–280 слов. Обязательно ≥1 разобранный кейс/пример.

### SKILLS — Навыки (владеть)
- Работа с информацией (анализ, синтез, интерпретация данных).
- Критическое мышление и аргументация.
- Способность к инновационным решениям.
- Владение методами управления (проектное / стратегическое / управление изменениями).
- **Объём:** 120–200 слов. Структура: список 4–6 навыков, каждый с 1–2 предложениями раскрытия.

### CONCLUSION — Заключение
- Подведение итогов с маркером «Таким образом, …» / «Итак, …».
- Выводы по поставленной проблеме.
- Рекомендации или перспективы дальнейшего исследования/решения.
- **Объём:** 80–150 слов.

### EXTRA — Дополнительные элементы
- Текстовое описание схем/таблиц/графиков (если уместно).
- Сравнительный анализ (если тема предполагает сравнение).
- Междисциплинарный подход — связь с экономикой, правом, социологией.
- **Объём:** 80–150 слов.

## Запрещённые артефакты в `expected_content`

- Префиксы вида `**Пример N.**`, `**Фрагмент N.**`, `**Основное содержание.**`, `[Текст следует добавить]`, `…`.
- Повтор фрагментов между блоками (CONCLUSION ≠ обрезок INTRO).
- Markdown-разметка внутри (звёздочки, заголовки, эмоджи).
- Нерелевантные теме термины и факты.

## Критерии приёмки билета

1. Все 6 блоков непустые, `is_missing=False`, `confidence=1.0`.
2. Объём каждого блока в рекомендованном диапазоне ±20 %.
3. Типизация: THEORY действительно содержит теорию и НПА; PRACTICE — конкретные кейсы; и т. д.
4. Ноль запрещённых артефактов из списка выше.
5. Spot-check 10 случайных билетов глазами не вызывает возражений у пользователя.

## Дизайн пайплайна

### 1. Источники для генерации

- `tickets.title`, `section.title`, `tickets.canonical_answer_summary`.
- Исходный фрагмент из PDF по позиции в TOC (если есть и непустой).
- Существующий контент блоков как «черновик» (для билетов с реальным телом в PDF).
- Уровень сложности `tickets.difficulty`.

### 2. Промпт Claude (Sonnet 4.6)

- **System:** «Ты — эксперт-методист, готовящий эталонные ответы на «отлично» по госэкзамену ГМУ. Структура ответа — строго по методичке МДЭ ГМУ. Каждый блок жёстко типизирован. Без шаблонных префиксов, без markdown в тексте.»
- **User:** заголовок билета + раздел + краткое summary + (опц.) исходный фрагмент + спецификация структуры из этого документа + критерии запрета артефактов.
- **Output:** JSON со схемой `{"intro": {"title": str, "expected_content": str}, "theory": {...}, ...}`.
- **Prompt caching:** system + спецификация структуры (≈4k токенов) кэшируются между вызовами.

### 3. Скрипт `scripts/regenerate_tickets.py`

```bash
python scripts/regenerate_tickets.py \
    --ticket-id tkt-002-...        # одиночный пилот
    --section sec-...              # по разделу
    --limit 5 --dry-run            # выборочный прогон
    --backup                       # создать бэкап БД
    --model claude-sonnet-4-6      # default
```

Поведение:

1. Backup БД в `exam_trainer.db.pre-claude-regen-<timestamp>`.
2. Выборка билетов по фильтрам.
3. Для каждого билета вызов Claude API с retry при ошибке.
4. Запись блоков в `ticket_answer_blocks` (UPDATE existing or INSERT).
5. Лог в `logs/regen-<timestamp>.jsonl`: `{ticket_id, status, tokens_in, tokens_out, ms, error?}`.
6. После прогона — автоматическая валидация на критерии приёмки и spot-check-отчёт.

### 4. Контроль качества

- **Auto-validation** после генерации: проверить объёмы, отсутствие префиксов-артефактов, маркер «Таким образом» в CONCLUSION, наличие НПА в THEORY (regex `Ф[ЗКП][\s-]\d+`).
- **Spot-check report:** случайные 10 билетов в `regen-spot-check-<timestamp>.md` — для просмотра пользователем перед мерджем.

### 5. Затраты

- 208 билетов × ~3000 input tokens × ~3000 output tokens.
- Sonnet 4.6: $3 / Mtok input, $15 / Mtok output.
- ≈ $2 input + $10 output = **~$12 за полный прогон**.
- Время: ~15–25 секунд на билет → 50–90 минут полный прогон последовательно, или 10–15 мин при batch=5.

## Зависимости и совместимость

- Не ломает существующие режимы — структура `ticket_answer_blocks` не меняется.
- `application/ticket_quality.py`-эвристика после регенерации даст 0 weak (все блоки полноценные) — 🔶 уйдёт сам.
- Понадобится новая зависимость: `anthropic` в `requirements.txt`.
- Конфиг: `ANTHROPIC_API_KEY` в `.env` (отдельный пользовательский ключ, не Claude Code).

## Что нужно от пользователя

1. **API-ключ Anthropic** (свой, не CC) — без него прогон 208 билетов невозможен.
2. Подтверждение спецификации эталона (этого документа).
3. Ревью пилота на 1 билете.
